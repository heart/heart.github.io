# รู้ไหม ทำไมโลกการเขียนโค้ดจะเข้าสู่ Contract-Driven Architecture

ก่อนจะไปถึงจุดนั้น มันไม่ได้เกิดขึ้นเพราะมีใครคิด framework นี้ขึ้นมาก่อน
แต่มันค่อย ๆ ไหลมา จากวิธีที่เราพยายามใช้ AI เขียนโค้ดนี่แหละ

ช่วงแรก เราใช้ AI แบบง่าย ๆ
เหมือนถาม StackOverflow เวอร์ชันเร็วขึ้น
ถามมันทีละฟังก์ชันให้ช่วยคิด logic หรือแก้ bug จุดเล็ก ๆ

ตอนนั้นมันยังไม่มีปัญหาอะไรใหญ่
เพราะคนยังเป็นคน "ถือภาพรวมของระบบ" อยู่
AI แค่ช่วยเร่ง

ถึงมันจะเขียนไม่ตรง naming บ้างหรือ style ไม่เหมือนในโปรเจ็ค เราก็แค่ copy มาแก้ๆ แล้วก็จบ
มันยังเป็นโลกที่
คนคิด -> AI ช่วย -> คนประกอบ

---

## พอเข้าสู่ยุค agent

พวก Claude Code, Codex อะไรพวกนี้เริ่มมา
มันไม่ใช่แค่ช่วยเขียนแล้ว

มันเริ่ม `อ่าน repo` เริ่ม `พยายามเข้าใจระบบ` แล้วก็เขียนโค้ดให้เราเลยทั้งก้อน

ตอนแรกมันรู้สึกดีมากนะ เหมือนมี junior ที่อ่านโค้ดเป็นแล้ว
แต่พอใช้จริงไปสักพัก จะเริ่มรู้สึกแปลก ๆ

โปรเจ็คมันใหญ่เกิน การให้มันไล่อ่านทั้งระบบ มันทำได้…แต่ไม่จริง

มันอ่านได้บางส่วนจำได้บางช่วง แล้วก็ "สรุปเอาเอง" ในหัวมัน
มันไม่ได้มี mental model ของระบบแบบที่คนมี

---

## แล้วปัญหาจะเริ่มโผล่แบบนี้

มันเขียนโค้ดที่ "ดูถูกต้อง" แต่ไปขัดกับอีก service หนึ่ง
มันใช้ abstraction ผิด layer แต่ดูผิวเผินไม่มี error
มัน bypass logic บางอย่าง ที่ไม่มีใครเขียนไว้ชัดๆ

หรือบางที มันแก้โค้ดให้เรา แล้วดันไป refactor ส่วนอื่นเพิ่มเอง

เหมือนมันพยายาม "จัดระบบให้ดีขึ้น" แต่จริง ๆ แล้วมันกำลังทำให้ system drift

---

## จุดนี้แหละที่คนเริ่มรู้สึกว่า

เอ๊ะ
ปัญหาไม่ใช่ AI เขียนไม่เก่ง ปัญหาคือ
**_มันไม่รู้ว่าอะไรห้ามทำ_**

---

# แล้วสิ่งที่คนทำโดยไม่ได้นัดกันคือ

เริ่มเขียนเอกสาร

ตอนแรกก็แค่:

- อธิบายว่า service นี้ทำอะไร
- input / output เป็นยังไง
- flow เป็นยังไง

แล้วพอเอาไปให้ AI อ่าน จะรู้เลยว่า
มันเขียนได้ดีขึ้นแบบชัดเจน

**_ตรงนี้หลายคนจะคิดว่า "โอเค งั้นก็แค่ต้องมี doc ดี ๆ"_**

แต่ถ้าใช้ไปอีกหน่อย จะเริ่มรู้ว่า
doc ธรรมดามันยังไม่พอ

---

# เพราะสิ่งที่พังจริง ๆ ไม่ใช่ logic ธรรมดา

มันคือพวกนี้

- tenant หายกลางทาง
- auth ไม่ถูก enforce
- data บาง field หลุดออกไป
- flow ที่ควร async กลายเป็น sync
- service ไปเรียกของที่ไม่ควรเรียก

พวกนี้มันไม่ใช่ bug แบบเขียน if ผิด
มันคือ `กฎของระบบ` ที่ถูกละเมิด

---

## แล้วคุณจะเริ่มเขียน doc แบบใหม่โดยไม่รู้ตัว

ไม่ใช่แค่เขียนว่า "อะไรคืออะไร"

แต่เริ่มเขียนว่า

- อันนี้ห้ามทำ
- อันนี้ต้องผ่านตรงนี้ก่อน
- อันนี้ต้องมี field นี้เสมอ
- อันนี้ห้ามออกไป public

ตอนนี้มันไม่ใช่คำอธิบายแล้ว
มันเริ่มเป็น `กฎ`

---

แล้วพอมีสิ่งนี้เกิดขึ้น flow มันจะเปลี่ยนเองเลย
จากเดิมเขียนโค้ดก่อน แล้วค่อย review กลายเป็น

**_เขียนกฎก่อน
แล้วค่อยให้ AI เขียนโค้ดตาม_**

---

ลองนึกภาพเคสง่าย ๆ
คุณอยากเพิ่ม feature export report

ถ้าไม่มีอะไรเลย
AI ก็จะ:

- ดึง data
- format
- expose endpoint

จบ

แต่ในโลกจริง
คุณรู้อยู่แล้วว่า:

- ต้อง filter tenant
- ต้องไม่ expose field บางตัว
- ต้อง log ทุกครั้ง
- ต้องไม่ block system
- ต้อง limit request

**_ถ้าคุณไม่เขียนสิ่งพวกนี้ไว้ AI จะไม่รู้_**

และมันจะเขียนโค้ดที่ "ถูก"
แต่ "ผิดในสิ่งที่สำคัญที่สุด"

---

## สุดท้ายคุณก็ต้องกลับมานั่งเขียนเองว่า

อะไรห้ามพัง
อะไรต้องมี
อะไรห้ามข้าม

ไม่ว่าคุณจะใช้ AI แค่ไหน
งานนี้คุณหนีไม่พ้น

แล้วตรงนี้แหละ ที่มันเริ่มชัด

สิ่งที่เรากำลังเขียนอยู่
มันไม่ใช่ documentation

มันคือ "contract"

---

## contract ในที่นี้ ไม่ใช่แค่ API

แต่มันคือข้อตกลงของทั้งระบบว่า

- อะไรคือ boundary
- อะไรคือ invariant
- อะไรคือสิ่งต้องห้าม
- อะไรคือสิ่งที่ต้องเกิดเสมอ

พอมี contract ดีพอ จะเกิดสิ่งแปลกอย่างหนึ่ง

AI จะเขียนโค้ด "ตรงมาก"

แทบไม่ต้องแก้
ไม่หลุด layer
ไม่ละเมิดของสำคัญ

แต่ถ้า contract ไม่ชัด

AI จะเดา
แล้วเดาผิดในจุดที่คุณไม่ได้เขียนไว้

---

## สุดท้ายมันเลยค่อย ๆ กลับด้าน

เมื่อก่อน
code คือของจริง
architecture คือคำอธิบาย

ตอนนี้
contract กลายเป็นของจริง
code เป็นแค่สิ่งที่เติมเข้าไป

---

## และพอทำแบบนี้ไปสักพัก เราจะเริ่มเห็นอีกเรื่องหนึ่งชัดขึ้น

สิ่งที่กำลังเกิดขึ้น ไม่ได้มีแค่การเปลี่ยนเครื่องมือ แต่กำลังสร้าง “ทักษะใหม่” ขึ้นมาด้วย
นั่นคือทักษะแบบ Contract-Driven

# Contract-Driven Skill เป็นทักษะใหม่ที่กำลังเกิดขึ้น

เมื่อก่อน คนเก่งอาจวัดกันที่การออกแบบระบบเก่ง เขียนโค้ดเก่ง หรือ review เก่ง
แต่พอ AI เข้ามารับงานเขียนมากขึ้น สิ่งที่มีค่าขึ้นมาทันที คือความสามารถในการ “นิยามกรอบของระบบ” ให้ชัดพอ จนไม่ต้องอาศัยการเดา

## ทักษะนี้จึงไม่ใช่แค่การเขียนเอกสารให้ละเอียด

- แต่มันคือการแปลงความเข้าใจเชิงสถาปัตยกรรม ให้ออกมาเป็นข้อตกลงที่ใช้ได้จริง
- บอกได้ว่าอะไรคือขอบเขต อะไรคือสิ่งที่ต้องเกิดเสมอ อะไรคือสิ่งต้องห้าม และการเปลี่ยนแปลงครั้งนี้ควรหยุดอยู่ตรงไหน

พูดอีกแบบ มันคือทักษะของการทำให้ architecture ไม่ได้อยู่แค่ในหัวคน
แต่กลายเป็นกฎที่ทั้งมนุษย์และ AI ใช้ร่วมกันได้

และนี่คือเหตุผลที่มันเป็นทักษะใหม่
เพราะในอดีต ต่อให้เอกสารไม่ชัดมาก ทีมก็ยังพอเดินต่อได้ ด้วยการอาศัย senior engineer, code review, และความเข้าใจร่วมกันในทีม
แต่ในโลกที่ AI เขียนโค้ดได้เร็วมาก ช่องว่างเล็ก ๆ ที่ไม่ถูกอธิบายไว้ จะไม่ถูกปล่อยว่างอีกต่อไป มันจะถูก AI เติมทันที

ดังนั้น คนที่มีค่ามากขึ้นเรื่อย ๆ จะไม่ใช่แค่คนที่ implement ได้เร็ว
แต่คือคนที่กำหนด contract ของระบบได้ชัด

และนั่นก็พาเรากลับมาที่คำถามตั้งต้น
ว่าทำไมโลกการเขียนโค้ดจะเข้าสู่ Contract-Driven Architecture

คำตอบไม่ได้ซับซ้อนนัก
มันเป็นเพราะเมื่อความเร็วในการสร้างสูงขึ้น การควบคุมด้วยความเข้าใจแบบกระจายอยู่ในคนอย่างเดียวมันไม่พออีกแล้ว
สิ่งที่เคยอาศัยการรู้กันเอง ต้องถูกเขียนออกมา
สิ่งที่เคยเป็นสามัญสำนึกของทีม ต้องถูกแปลงเป็นข้อตกลงของระบบ

ท้ายที่สุดแล้ว โลกไม่ได้จะเข้าสู่ Contract-Driven Architecture เพราะมันดูสวยหรือเป็นแนวคิดใหม่ที่น่าตื่นเต้น
แต่เพราะมันกำลังกลายเป็นวิธีเดียวที่ทำให้เราใช้ AI เขียนโค้ดได้มากขึ้น โดยที่ระบบยังไม่ค่อย ๆ สูญเสียรูปทรงของมันไปทีละน้อย
